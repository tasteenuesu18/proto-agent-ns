# 実装計画：1問1答式エージェントプロセス可視化ダッシュボード (追加修正)

## 概要
ユーザーの要望に従い、対話型のチャットUIから「1問1答形式でのエージェント思考プロセス可視化ダッシュボード」へとアプリケーションの構造とUIを根本的に改修しています。
今回、追加の要望として「LLM（脳）」と「エージェント（手足）」の役割境界を視覚的に明確に分離すること、API呼び出しの安定化、入力制限を追加します。

## 要件の追加・変更点
1. **get_weatherツールのAPI見直し**
   - 現在のOpen-MeteoやGeocodingの組み合わせで結果がうまく返っていない問題を解消するため、よりシンプルにレスポンスが取得できる天気API（例: `wttr.in` のJSON形式など）に変更します。
2. **アクター（LLMとAgent）の視覚的分離表示**
   - **実行シーケンス概要**: それぞれのステップが誰(User, Agent, LLM)によって行われているかをアイコンとラベルで明記します。ツールの実行は「Agent（システム）」が行い、テキストやツール呼び出しの指示は「LLM」が行うという境界を明確にします。
   - **詳細ダッシュボードの表示**: LLM送受信ログをより詳細にし、「AgentからLLMへ何が送信されたか（プロンプト等）」「LLMからAgentへ何が返されたか（推論結果、ツール実行要求など）」を明確にタブ分けまたはセクション分けして可視化します。
3. **ユーザー入力の制限**
   - 自由入力(テキストボックス)を廃止し、あらかじめ用意されたテンプレート(サジェストボタン)からの選択のみでリクエストが飛ぶように制限します。

## 提案するシステム構成と変更点

### 1. APIエンドポイントの改修 (`src/app/api/chat/route.ts`)
- `get_weather` ツールの `execute` 処理を修正し、確実にレスポンスが返る形に実装し直します。
- `onStepFinish` イベント等で取得しているLLMとのやり取り情報を、UIで「送信(リクエスト)」「受信(レスポンス)」として分離して描画しやすい形に整理します。
- 複数ステップの情報を分かりやすくするため、AgentとLLMの境界を意識したデータ構造（必要に応じて `requestBody` のキャプチャ等）を追加します。

### 2. UIデザインとコンポーネントの改修 (`src/app/page.tsx`)
- **左ペイン**: 
  - 入力フォーム（`<input>`）を取り除き、サジェストボタン群のみを残します。
  - シーケンスタイムラインのデザインを修正し、`[User]`, `[LLM]`, `[Agent]` というアクターが視覚的に分かるように構造化します。
- **右ペイン**: 
  - 「タブ2: LLM送受信ログ」を「AgentからLLMへの送信 (Prompt)」「LLMからAgentへの返答 (Generation)」といったように役割ごとに表示領域を分けます。

### 3. Verification Plan
- `wttr.in` などを使用した天気情報の取得が正常に動作するか（「東京の天気を教えて」のサジェストで動作確認）。
- 各ステップの実行シーケンスや詳細ログに、アクターの境界（LLM vs Agent）が正しく反映されているか確認。
- 自由入力欄がなくなり、サジェストクリックのみで処理が完結することの確認。
